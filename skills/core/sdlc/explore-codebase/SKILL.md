# explore-codebase — ground the design in what already exists

Before a line of design is written, study the **real code** the feature will extend and write
a **codebase map**: the flows it touches, the *distinct execution modes* each of those flows
has, and the constraints/patterns the design must respect. This exists to stop the HLD being
invented from the prompt alone — the single most common design failure is choosing an
architecture that the happy path allows but an existing mode (multi-turn, async, batch,
streaming) forbids. This is a read-only investigation artifact, not code and not a design —
never edit app code, never design APIs or schemas here.

## Inputs
Your instructions name what to read — the requirement folder and the feature summary — and the
artifact path to write. Read every requirement file plus each repo's `CLAUDE.md`/`AGENTS.md`
first. Standalone (`/explore-codebase`)? Work from the feature description the user gives you
and write the map to a sensible path you choose (and tell them where).

## Method
1. **Find the surface.** From the requirement, locate the feature area(s) in the code — the
   entry points (controllers/routes/CLI/handlers/UI screens), the services/modules behind
   them, the data model, and the tests/fixtures. Read routing → service/domain → persistence
   for each path that matters.
2. **Trace every flow end-to-end.** For each user-visible flow the feature touches, follow the
   real call path from entry to persistence/response. Do not stop at the first path — code
   almost always branches.
3. **Enumerate the execution modes.** This is the load-bearing step. List every DISTINCT mode
   / branch the flow runs in — e.g. single-turn vs multi-turn, sync vs async/queued, batch vs
   streaming, first-request vs idempotent-retry, single-tenant vs multi-tenant — and, for each,
   *what drives it* (which field/flag/config selects it, and where the logic lives). A design
   that handles only one mode is the failure this step prevents.
4. **Capture reusable patterns & constraints.** Existing APIs/DTOs to reuse, the auth/guard
   pattern, error envelope, idempotency/concurrency conventions, migration tooling, house
   style — only what actually constrains the design. Cite `file:line` for each.
5. **Flag the traps.** Anything that would break a naive assumption (e.g. "prompts can't be
   batch-fetched because turn N+1 is generated by the server from turn N's response"). Call
   these out loudly — they are the reason this document exists.

In an **umbrella / multi-repo** layout, repeat across every sibling repo the feature spans;
label each finding with its repo.

## The map must cover (write all)
1. **Feature surface** — the flows in scope and where they live (entry points, key modules),
   cited.
2. **Flow traces** — for each flow, the end-to-end call path (happy + main error paths).
3. **Execution modes** — the enumerated list from step 3: each mode, what selects it, where it
   is implemented, cited. This section is mandatory and must be exhaustive.
4. **Reusable patterns** — existing APIs/DTOs, auth, error handling, idempotency,
   persistence/migration conventions to build with, cited.
5. **Constraints & traps** — hard constraints and the naive-assumption traps from step 5, each
   with the evidence that establishes it.
6. **Repos touched** — which repo(s)/paths the feature spans (relevant in umbrella layouts).

## Standards
- **Evidence, not memory.** Every claim cites `file:line`. Never assert an API, table, mode, or
  behaviour you have not read. If you could not verify something, say so — an unverified guess
  here becomes a wrong architecture downstream.
- **Exhaustive on modes, terse elsewhere.** The value is completeness of the execution-mode
  list; keep the rest tight.
- **Investigate, don't design.** No chosen approach, no API shapes, no schemas — that is the
  HLD/LLD's job. You describe only what *is*.

## Safety
- Writes are limited to the map document at the path your instructions give. Never edit
  application code, and never touch `.maestro/runs/<slug>/state.yaml` or other run state.

## Output contract
Return `map_path` (where you wrote the map), `map_summary` (2–3 sentences: the flows covered
and the sharpest constraint the design must respect), and `modes_found` (a short scalar — the
count or a compact comma-separated list of the distinct execution modes you enumerated, e.g.
`3: single-turn, multi-turn, iterative`). The map file is the single source of truth; the
scalars are only for routing and orientation.
